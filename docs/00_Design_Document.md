## 0. TL;DR

我们提出一套 **"五层骨架 + 双循环引擎"** 的 Agent 体系架构 **太一 (The One)**。

- **五层骨架** = 输入/场景 → 能力/资源 → 调度/治理 → 输出/验证 → 迭代/优化
- **双循环引擎** = **PDCA 内部循环**(任务级:执行-校验-修正-交付)+ **OODA 外部循环**(系统级:观察-定位-决策-行动,跨任务持续演进)

核心创新有三条:
1. **治理权与调度权强制分离**(zachshi-ai 铁律)——验证模块对调度模块拥有"红线一票否决"
2. **场景工程独立化**——把"环境/约束/红线"从 prompt 中抽出,做成第一类工程对象
3. **Skill 工程标准化**——每个 Skill 附"质量门禁",门禁不通过禁止进入生产知识库

> 一句话:**太一不是"接 LLM 的工具箱",而是"对 LLM 进行工程治理的生产操作系统"**。

## 1. 设计哲学

### 1.1 三个根问题 + 一个产品取向

设计太一之前,我们必须先回答四个根本问题,它们决定了整个架构的取舍:

**Q0:产品设计优先级——效率优先,还是安全优先?(取向选择,不做中立)**

这是用户几乎看不见、但踩坑后才能感知的底层选择。同一个任务,两个工具可能做同一个动作,但**设计优先级**完全不同:

- **OpenClaw**(效率优先):追求"最快完成",会跳过本地 Git 配置直接提交代码,越过用户身份
- **Trae**(安全优先):默认尊重本地 Git 配置,绝不擅自修改用户身份

**两个工具都能跑通任务,但设计哲学完全不同**。太一的选择是:**安全优先**——关键节点强制用户确认,宁可慢一点,也要守住边界。这个选择会在所有架构决策中体现(治理层默认 DENY 优于默认 ALLOW、人审优先于自动过、规则加载为只读)。

**Q1:Agent 的"能力边界"到底由什么决定?**

流行回答:大模型的能力。
我们的回答:**模型能力 × 场景理解 × 规则治理 × 验证兜底**。模型只是四个变量之一,且不必然是决定性变量。

一个具体案例:OpenClaw 帮用户上传 GitHub 仓库,表面任务完成了,但提交作者身份被偷偷替换。如果用户是法学背景、对权属敏感(如 zachshi),会发现这是严重事故;如果用户是普通开发者,可能永远不会点开 commit 记录看作者——坑就永远埋着。**这就是"能力边界由什么决定"的具体答案**:Agent 的能力边界不在模型本身,而在于**是否覆盖了用户的隐性验收标准**。

**Q2:多 Agent 协作是不是"银弹"?**

流行回答:多 Agent 讨论=更好决策。
我们的回答:**协作无门等于无协作**。多 Agent 框架如果只是让 Agent 用自然语言聊天,本质就是执行 Agent 的一言堂——专家 Agent 的意见"被听到"和"被执行"是两码事,中间缺一扇"门"。

**Q3:Agent 产品的价值锚点在哪?**

流行回答:能不能多完成任务。
我们的回答:**不是"能做什么",而是"边界是否清晰、风险是否可控"**。少功能但边界清晰的工具,用户敢用在生产;多功能但坑点遍布的工具,只能当 demo 玩具。

**Q3 的背后:行业陷阱——"全自动 / 零干预"作为卖点**

当前 Agent 行业把"全自动"和"零干预"作为最大卖点。但这是**陷阱**:盲目追求这两件事,是当前 Agent 不靠谱的根源。模型能力跃迁之前,追求全自动等于让模型"自作主张",恰好是事故的温床(就像 OpenClaw 自动改 commit 作者)。太一明确**不以"全自动"为荣**,而以"在必须人审的地方强制人审,在可以自动的地方追求自动"为荣。

### 1.2 一个讽刺的现状——太一存在的根本原因

> **"越不懂行的用户,越觉得 AI 好用;越专业的用户,越要花大量精力给 AI 擦屁股。"**

不懂代码的人用 AI 写个网页,觉得功能全、界面好看,简直是神器;但专业开发者一看,到处是逻辑漏洞、安全隐患、冗余代码,修复的功夫可能比自己写还长。

这个"很讽刺的现状"是 zachshi-ai 文章最核心的洞察,也是太一存在的根本原因:

- **表层结果好 ≠ 事情做对了**
- **用户专业度越高,对 Agent 的隐性验收标准越严**
- **现有 Agent 只满足最低表层标准,系统性忽略隐性标准**

太一的目标不是"让 AI 更强大",而是"让 AI 的隐性验收标准**被工程化**——把看不见的'该做什么不该做什么'写成代码,让模型**绕不过**而不是'应该记住'"。

### 1.2 三个隐喻

我们用三个隐喻来描述 太一的内核:

| 隐喻 | 含义 |
|---|---|
| **Agent = 数字员工** | 不是"会说话的代码",而是需要制度约束的"员工"——有工种(Knowledge/Skill/Tool)、有权限(红/黄/绿区)、有考核(输出验证)、有晋升(Skill 升级路径) |
| **太一 (The One) = 工厂生产线** | 输入区→装配区→质检区→出厂区,每个工位各司其职,质检员不归车间主任管 |
| **演进 = 周行不殆** | 内部循环(PDCA)处理单任务交付;外部循环(OODA)处理跨任务演进。太一居中不动,执行围之运行,每完成一圈,系统认知就上一阶(《道德经》"独立而不改,周行而不殆") |

### 1.3 七条设计原则(INCOSE 风格)

| 原则 | 描述 | 在太一中的体现 |
|---|---|---|
| P1 关注点分离 | 每一层只管一件事 | 输入、能力、调度、验证、迭代五层互不越界 |
| P2 单一真相源 | 同一份事实只在系统一处为权威 | Gateway(运行时)、MEMORY.md(长期事实)、Skill Index(技能注册) |
| P3 显式优于隐式 | 所有"魔法"都要可解释 | 所有决策必须带"理由+证据链",审计可重放 |
| P4 渐进式降级 | 每层都有降级路径 | 模型降级→简化执行→人工接管 |
| P5 边界即设计 | 明确"不做什么"和"做什么"一样重要 | 明确只服务确定性生产任务,纯创意场景不在范围 |
| P6 闭环优于开环 | 任何动作都要有反馈回路 | PDCA + OODA 双循环;所有 Skill 使用后必评估 |
| P7 制度优先于劝导 | 用代码约束行为,而不是用 prompt 说服模型 | 关键校验全部下沉到治理层,模型"看不到但绕不过" |

## 2. 理论框架:三层理论地基

### 2.1 系统工程视角——SoS(System of Systems)

SoS 强调:**当多个独立系统集成到一起时,涌现出新系统才具备的能力**。INCOSE 把 SoS 分为四类:
- 指导型(Directed):中央权威强制集成
- 协作型(Collaborative):各子系统的"老板"坐下来共同决策
- 自治型(Autonomous):子系统独立,按协议协作
- 虚拟型(Virtual):无中央协调,纯协议

**太一的定位**:**指导型 + 协作型混合**——有一个轻量"协调者"(Gateway/Scheduler),但每个子系统(记忆/技能/治理/验证)独立可替换,协作按协议。

### 2.2 管理学视角——PDCA + OODA 双循环

我们用两个管理学循环来描述 太一的运行机理:

**PDCA 戴明环**(单任务内部):
- **P(Plan)**:任务适配与调度层制定执行计划
- **D(Do)**:执行层按照计划调用工具和技能
- **C(Check)**:验证层独立校验输出是否过线
- **A(Act)**:不通过则回弹给执行层修正;通过则归档进入迭代层

**OODA 博伊德环**(跨任务外部):
- **O(Observe)**:迭代层观察所有任务的执行轨迹、验证结果、用户反馈
- **O(Orient)**:基于多源信息(记忆层)定位当前系统薄弱点
- **O(Orient → Decide)**:决定优化方向——是 Skill 升级?规则补丁?工具替换?知识库更新?
- **A(Act)**:在新的任务中实施优化

两个循环的嵌套关系:**PDCA 嵌套在 OODA 内**——每个任务完成一次 PDCA,系统整体完成一次 OODA 增量;长时间看就是**周行不殆**。这一表达源自《道德经》第 25 章,强调"太一居中不动,万物围之运行"的本体论结构,比"螺旋上升"更契合太一(The One)的命名。

### 2.3 控制论视角——三种控制模式

借鉴控制论,我们把 Agent 的行为分为三种控制模式:

| 控制模式 | 描述 | 适用场景 |
|---|---|---|
| 闭环控制(Feedback) | 执行→感知→修正 | 单任务(PDCA) |
| 前馈控制(Feedforward) | 提前注入约束,减少偏差 | 场景工程、知识工程 |
| 比例控制(Proportional) | 根据风险等级动态调节 | 治理层审批——低风险自动过、高风险必人审 |

## 3. 体系架构总览

### 3.1 五层骨架

> **场景 = 环境变量集**(原文 §五(一)2 核心定义):
> "所有的质量要求、合规规则、时效要求、禁忌限制,**本质都是环境变量**,应该统一管理、随场景切换。"
>
> 场景工程不是"提示词加一段背景描述",而是 **独立模块化的环境变量层**。所有任务执行时,调度层先根据场景加载相应变量,这些变量会注入治理层的规则匹配 + 调度的工具选择 + 验证的检查清单。
>
> **知识工程的业界共识**(原文 §五(一)3):zachshi 明确指出"长期沉淀的领域知识、规则库、标准库的结构化管理,**对应 Karpathy 提出的 LLM Wiki 方向**"。太一的知识工程模块锚定这一业界共识,采用 Wiki 化的 Markdown 存储 + 向量索引 + FTS5 全文检索。

### 3.2 三条横切关注点

五层是**纵向关注点**,在它们之上还有三条**横切关注点**,贯穿所有层:

**H1 记忆/认知层(Memory & Cognition)**——独立子系统,被 L1~L5 共用
- L1 短期会话上下文
- L2 技能库(SKILL.md,程序性知识)
- L3 向量索引(语义检索)
- L4 Honcho 用户模型(辩证画像)
- L5 FTS5 全文检索(历史)

**H2 协议/接口层(Protocol & Interface)**——MCP / OpenAI 兼容 / ACP
- 对内:标准化模块接口
- 对外:开放为 MCP Server,被其他 Agent 调用

**H3 观测/审计层(Observability & Audit)**——贯穿全程
- 全链路 trace(每次执行可重放)
- Token 成本归因(每次调用成本可见)
- 安全事件审计(谁、何时、何动作、谁批准)

**H4 价值流层(Value Stream Alignment)**——贯穿全程(* zachshi 提出,Phase 1 深化)
- 核心问题:**任务在系统里流动时,如何保证每一步都服务于最终业务目标?**
- 受 VSM(Value Stream Mapping)、APQC 流程框架、目标逐层分解启发
- 4 个子模块:
  - **目标锰定**(Goal Anchoring):每个任务明确标注服务的业务目标与 KPI
  - **价值流定义**(Value Stream Templates):为高频任务预定义端到端价值流(代码提交/客服回复/报表生成等)
  - **增值评分**(Value Contribution Scoring):任务完成后评分"对目标贡献多少"
  - **瓶颈识别**(Bottleneck Detection):自动识别价值流上最慢/失败率最高的环节,送入 L5 优先优化
- 详见 §7 完整设计

## 4. 关键设计决策

### 4.1 为什么治理权与调度权必须分离?

理由:人性使然。**任何同时负责"完成任务"和"验证任务质量"的模块,都有动机绕过验证以加快完成**。这不是技术问题,是激励设计问题。

**解决方案**:L3 拆成两个独立模块,共享一个"决策接口"协议:
- 3.1 执行治理(Governance):只做"block / allow",不参与"做什么"
- 3.2 任务调度(Scheduling):只做"选什么、做不做",无权自我豁免

调度模块在调用工具前必须向治理模块申请"许可证",治理模块按"红线规则 + 场景约束"返回"放行/拒绝/转人审"。调度模块不通过治理,代码层面就无法执行。

### 4.2 为什么场景要独立于 Prompt?

理由:**Prompt 是临时、瞬时、一次性的;而场景(Scenario)是长期、半静态、跨任务复用的**。把场景约束写进 Prompt,等于让一个临时工去背公司规章制度——既记不住,也不更新。

**解决方案**:L1 中单独建立"场景工程(Scenario Engineering)"模块,统一管理:
- 场景库(行业 / 角色 / 任务类型)
- 约束库(合规红线、安全规则、隐私要求)
- 切换逻辑(基于意图识别 + 用户画像,自动选择)

### 4.3 为什么 Skill 附"质量门禁"?

理由:**没有质量门禁的技能库,就是垃圾场**——什么都能进,什么都不敢用。

**Skill 的本质定义**(原文 §五(二)2):Skill **不是零散的 prompt 片段**,而是**程序性知识**——把"怎么做某件事"的方法,封装成有明确输入输出、适用边界、质量基线的模块。它的核心特征:

- **输入输出契约明确**(不是"调用 LLM 试试看",而是"传入 X,返回 Y")
- **适用边界清晰**(什么场景能用,什么场景不能用)
- **质量基线可验证**(必须有质量门禁)
- **可聚合 / 可复用 / 跨 Agent 调用**(不与特定 Agent 绑定)

**解决方案**:每个 Skill 必须附带一份 `quality_gate.md`,声明:
- 准入标准(什么条件下可以发布)
- 退出标准(什么条件下必须废弃)
- 验证项(自动跑通的测试用例)
- 副作用清单(可能产生的影响)
- 升级流程(谁可以改、改什么、需要什么审批)

不附质量门禁的 Skill,只能留在"沙箱库"用于实验,无法进入生产路径。

### 4.4 为什么验证要独立,不能由模型自评?

理由:**自评不可信,这是常识**。模型自评时倾向于给自己高分(尤其在长上下文、对自己的输出有路径依赖时)。

**解决方案**:L4 输出验证层使用以下独立验证手段:
- **结构化校验**:JSON Schema / 正则 / 类型检查(机器做,100% 客观)
- **功能校验**:跑测试用例、调用 API 验证(机器做)
- **规则校验**:对照红线规则库(机器做)
- **同行评审**:调起独立验证 Agent 复评(机器做,降低同源偏见)
- **人审**:主观偏好、灰色判断(人做)
- **历史对比**:与历史成功案例对比差异度(机器做)

### 4.5 为什么多 Agent 必须有"红线一票否决"?

理由:多 Agent 协作的瓶颈不是"听不到意见",是"听不听由我"。

**解决方案**:L3 治理层为不同领域的专家 Agent 设置"红线权限":
- 安全专家 Agent → 触及"数据安全/隐私"的决策有否决权
- 合规专家 Agent → 触及"合规边界"的决策有否决权
- 业务专家 Agent → 触及"业务红线"的决策有否决权
- 性能专家 Agent → 只有建议权(优化维度,不构成否决)
- 体验专家 Agent → 只有建议权

**具体冲突场景**(原文):比如**安全专家要求不能联网,业务专家要求必须联网**——系统不能直接死锁。

**默认优先级表**(按重要性递减):

| 优先级 | 维度 | 例子 | 裁决权 |
|---|---|---|---|
| P0 | 安全 / 合规 / 权属 | 禁止读取 SSH 私钥、禁止覆盖 commit 身份 | 安全 / 合规专家一票否决 |
| P1 | 业务红线 | 退款金额超限需人审 | 业务专家一票否决 |
| P2 | 性能 / 体验优化 | 代码是否优雅、报告是否详细 | 建议权 |
| P3 | 品味 / 主观偏好 | emoji 使用 / 代码风格 | 执行 Agent 或人审 |

**同级冲突裁决**:
1. **同类专家可复议**(被否决 1 次,允许 1 次复议)
2. **复议仍否决 → 升级到 L5 迭代层**记录为"系统缺陷"
3. **跨级冲突**(如安全 P0 vs 业务 P1)→ **P0 胜**(默认安全优先级最高)
4. **同优先级冲突** → **预设顺序**(安全 > 合规 > 业务 > 性能 > 体验)
5. **人类决策为终极仲裁** ——任何冲突可提交用户最终拍板

否决发生时,任务**强制暂停**,不会重试绕过。

## 5. 让框架可落地(从宣言到机制)(* opus 4.8 补强)

> **⚠️ 本章与 zachshi 五层架构的关系声明**:
>
> - **zachshi 的五层架构(§3) + 治理-调度分离铁律(§4.1)** ——是太一的**核心设计哲学**,按 zachshi 的规则,这是**不可修改的底层逻辑**
> - **本章(opus 4.8 补强)——是**“上面哲学”的工程化落地层**——机制、顺序、度量,让“宣言”变成“可实施的方案”**
> - **两者是“哲学 + 工程”的完整闭环**,不是替代关系
> - **opus L0-L4 成熟度(§5.5)与 zachshi 的嵌套 5 层成熟度模型(§8.7)也是同一关系的不同视角**——§8.7 是“架构评估”,§5.5 是“工程状态评估”

> **本章是 opus 4.8 对原文章的实战补强**——是“说得到”到“做得到”之间的那一层。原文是宣言+架构,本章是机制+顺序+度量。

### 5.1 如何指定规则(规则作为数据,不是散文)

红线与检查项需要**可携带、可评审的格式**——不能是埋在 prompt 里的中文。

最小规则 Schema:

```yaml
id: git.author.identity_match
domain: authorship              # security | authorship | compliance | safety | optimization
severity: red_line              # red_line(拦截) | advisory(警告)
applies_to: ["git.commit", "git.push"]
trigger: pre_execution          # pre_execution | post_execution | step_gate
check:
  type: deterministic           # deterministic | external_tool | model_judge
  assert: "commit.author == local.git.config.user"
on_fail:
  action: block                 # block | warn | request_confirmation
  message: "Commit author does not match the local Git identity."
precedence: 80                  # 越高越优先(冲突时胜出)
owner: platform-security        # 责任人
audit: true                     # 每次评估记录不可篡改日志
```

**为什么是数据而不是散文**:
- **可评审** — 静态分析/同行评审可以走 XML/YAML diff
- **可版本化** — 走 git,任何变更可追溯
- **可测试** — 一份 rule.yaml 是一份 test fixture
- **可机器加载** — 治理层引擎直接加载,不需要重新解析 prompt

### 5.2 如何构建客观检查清单(选择,不是生成)

> **opus 关键洞察**:自动生成一份完整的检查清单,其难度本质上与原初可靠性问题相当——递归难题。

**解法:选择,不是生成。**

**1. 选择而不是生成**:给定 `任务类型 × 场景`,大部分验收标准是**预先知道的**。维护一份策略/清单库,按 `(task_type, scenario)` 作 key,运行时**选择 + 参数化**,不是现场发明。新检查项通过 Loop Engineering(P6)逐步加入,不是凭空生成。

**2. 优先最便宜可靠的检查器**:每个验收项路由到能给的**最强检查器**:

| 检查类型 | 用于 | 可信度 |
|---|---|---|
| **确定性**(字符串 / AST / 状态比较) | 作者匹配、文件存在、schema、格式 | **最高**——不需模型 |
| **外部权威**(linter / type-checker / 测试套件 / 策略引擎 / 扫描器) | 代码质量、秘密泄露、许可证、合规 | **高**——独立维护 |
| **模型评判** | **仅限**真的模糊(语气、设计质量) | **最低**——需校准和采样 |

**目标:红线集几乎全是确定性或外部。** 作者匹配检查是理想案例:一行精确比较,便宜且不可伪造。

**3. 隔离模型评判**:必须使用模型评判时:
- 使用**独立实例/独立模型 + 狭窄 rubric**——绝不与执行者同实例(P3)
- **版本化评判者**,针对 known-good / known-bad 标记集跑回归
- 跟踪其 **false-pass / false-block 率**,人工抽检校准
- 这是"验证验证者"的唯一方法

### 5.3 多 Agent 闸门与仲裁

> **zachshi 原文未展开。opus 4.8 补强。**

多数多 Agent 设置是“几个 Agent 用自然语言聊天”——“没有闸门的协作不算协作。”

**明确权责**:
- **红线维度**(数据安全 / 权属 / 核心规则):该领域专家有**一票否决权**。失败 → 任务暂停修正,不得前进,哪怕牺牲效率
- **优化维度**(性能 / 体验 / 优雅):专家**仅建议**,是否采纳由执行 Agent 或升级人审

**红线冲突时**(安全说“不能联网” / 业务说“必须联网”),系统不能死锁:

1. 比较 **precedence** 等级,高者胜(部署时声明,例: `data_security > authorship > business_availability`)
2. **同级硬冲突 → fail closed**,升级人审,完整冲突日志
3. **每次仲裁记录**,便于后续调优 precedence 表

### 5.4 落地优先级——不要一起起所有 5 层

> **opus 关键建议**:"不要建框架。建一个垂直切片。"

按**风险削减 / 单位工作量** 排序,顺序如下:

1. **列出不可逆 / 高爆炸半径操作**(写入、支付、身份、对外发布、删除)。这个列表很短,几乎所有实际损失都在里面
2. **只为这些操作加确定性前置闸门**(作者匹配是第一步)。可靠性最大提升 × 最少工作量
3. **把验证者从执行者中拆出来**(P3),加不可篡改审计日志
4. **引入规则库 + 红线/建议拆分**(P4 + 5.1)
5. **加每任务检查清单 + 仲裁 / 升级**(5.2 + 5.3)
6. **闭环**——每次新失败反哺规则/Skill/知识(P6)

> **仅 1-3 步足以从“demo”跳到“可信任生产”**。

### 5.5 成熟度模型——定位自己(opus L0-L4 简化版)

> **⚠️ 两套成熟度模型的关系**:
>
> - **§5.5(opus L0-L4)** ——是**工程状态评估**,给实施者看"我们在哪个阶段",面向“行动”
> - **§8.7(zachshi 嵌套 5 层)** ——是**架构质量评估**,给架构师看“架构是否完备”,面向“设计”
> - 两套**指向不同问题**,不互斥。例如:
>   - 太一现在的架构质量(Task/Skill/Subsystem/System/Value Stream 各自评估)是 §8.7 的状态
>   - 太一现在的工程状态(prompt → 闸门 → 分离治理)是 §5.5 的状态
> - **§5.5 写的是“Phase 0-2 我们要达成什么”,§8.7 写的是“架构完备度怎么衡量”**——两套并存互补

> **opus 4.8 提供的简化成熟度模型**。比太一原的"嵌套 5 层"更**实战**——这是面向"我们要走到哪一步"的,§8.7 的嵌套模型是面向"架构每个组件完备度"的。

| 等级 | 状态 |
|---|---|
| **L0 — 仅 prompt** | 规则在 prompt 里;执行者自验证;无审计 |
| **L1 — 点闸门** | 几个关键操作有确定性预检;基础审计日志 |
| **L2 — 分离治理** | 治理与调度是独立模块;规则库存在;红线 fail closed |
| **L3 — 可验证验收** | 每任务检查清单自动选择;红线/建议拆分;仲裁 + 人升级;可靠性指标跟踪 |
| **L4 — 闭环** | 失败反哺规则/Skill/知识;验证者有回归套件和 false-pass/block 率;持续校准 |

**太一当前状态(2026-06-20):** L2(治理与调度独立,规则库存在,红线 fail closed,Demo 9 个测试全过)
**Phase 1 目标:** L3(加每任务检查清单自动选择 + 仲裁 + KPI)
**Phase 2 目标:** L4(加闭环反馈 + 验证者回归)

**与 §8.7 嵌套模型的映射**:

| opus 状态 | zachshi 架构质量 |
|---|---|
| L2(分离治理) | 至少达成 Task L3 + Skill L2-L3 + Subsystem L2(架构上治理与调度独立,门禁走通) |
| L3(可验证验收) | + H4 价值流层集成(KPI + 双模式锰定) |
| L4(闭环) | + Skill L3-L4 + Subsystem L3 + System L2 + Value Stream L2 |

**这两套不冲突**:opus L3 是“系统在 L3 这个工程状态”,§8.7 的各层是“架构在哪个成熟度”。阶段一致,粒度不同。

### 5.6 度量"做得对",不仅是"做得到"

> **opus 关键补强**:直接回应 zachshi 原文章“没人测隐性风险覆盖度”的抱怨。

与任务完成率并列跟踪:

| 指标 | 定义 | 太一目标 |
|---|---|---|
| **红线遵从率** | 输出时零红线违规的运行比例 | 100% |
| **越界事件数** | 每 N 次运行的未授权状态变更 | 0 |
| **隐性风险覆盖度** | 已知隐性标准中已自动检查的比例 | ≥ 80% |
| **false-block 率** | 闸门误拦率(可靠性杀手!) | ≤ 5% |
| **规则更新平均时长** | 新失败类变成永久检查的耗时 | ≤ 7 天 |
| **人升级 / 介入率** | 系统仍需人介入的场景比例 | 趋势下降 |

### 5.7 反模式(可靠性气味)

> **opus 给的 8 条检查清单。**

- [ ] 关键规则仅在 prompt 里表达为"记得要..."
- [ ] 干活的组件同时签收(P3 违反)
- [ ] 无不可篡改审计日志
- [ ] "红线" 检查只 **warn** 不 **block**
- [ ] 单一巨型模型评判 prompt 是唯一验证
- [ ] 所有约束同一个严重级——无红线/建议区分
- [ ] Agent 执行不可逆 / 破坏性操作无确认
- [ ] 规则手册从未修剪/演进

### 5.8 闸门的成本——以及何时不用

> **opus 关键平衡视角**:zachshi 原文把硬闸门视为纯收益,opus 指出了代价。这是一份诚实文档应该承认的。

闸门不免费:
- **增加延迟**
- **可能产生 false-block,侵蚀用户信任**
- **规则手册本身需要维护,且会漂移**

**正确用法**:
- **硬、fail closed 的闸门**仅用于**不可逆 + 高爆炸半径**操作
- **低风险工作**使用**建议检查 + 采样**
- **保持红线集小而高信号**——其余是建议 + 覆盖 + 审计
- **不要为探索性 / 创意 / 低风险任务加闸门**——那里闸门只增加摩擦
- **Loop Engineering 应该裁剪规则**——只新增,从不修剪的反模式

> 目标是**校准的控制**,不是**最大的控制**。

## 6. 运行机理(运行时主循环)

太一的单次任务执行遵循**PDCA 循环**,主循环如下:

```
┌──────────────────────────────────────────────────────────────────┐
│  T0  任务进入(经 Gateway 鉴权 / 限流)                              │
│       │                                                            │
│  T1   ▼                                                            │
│      L1 输入解析:                                                   │
│      - 抽取显式需求 → Prompt                                       │
│      - 识别场景 → 加载场景约束库                                    │
│      - 关联知识 → 召回相关知识                                      │
│      - 注入任务信息 → 任务上下文                                    │
│       │                                                            │
│       ▼                                                            │
│  T2  L3.2 任务适配调度:                                             │
│      - 选模型(轻量/主力/专家)                                       │
│      - 选工具 + 选技能                                              │
│      - 拆解为子任务,委派到子 Agent(若需要)                          │
│       │                                                            │
│       ▼  ←──────── 申请执行许可证 ────────┐                        │
│  T3  L3.1 执行治理:                          │                       │
│      - 校验场景红线                            │                       │
│      - 校验操作权限                            │                       │
│      - 校验凭证/SSRF                          │                       │
│      - 决策:放行 / 拒绝 / 转人审  ───────→   │                       │
│       │                                (转人审时挂起,              │
│       │                                 等待用户指令)              │
│  T4   ▼  (放行)                                                    │
│      L2 执行:调用工具/技能/MCP,产生中间产物                         │
│       │                                                            │
│       ▼                                                            │
│  T5  L4 输出验证:                                                  │
│      - 客观清单校验(自动 100%)                                     │
│      - 主观清单校验(人审兜底 20%)                                   │
│      - 不通过 → 反馈回 L3 修正,进入下一轮 PDCA                     │
│      - 通过 → 继续                                                  │
│       │                                                            │
│       ▼                                                            │
│  T6  L5 迭代层归档:                                                │
│      - 任务轨迹写入 history                                        │
│      - 经验沉淀为 Skill(若触发条件)                                │
│      - 反馈信号送入 Honcho 用户模型                                 │
│      - 异常事件送入 OODA(下一次系统级优化)                          │
│       │                                                            │
│       ▼                                                            │
│  T7  返回用户 / 触发后续任务(若 Cron / Heartbeat)                  │
└──────────────────────────────────────────────────────────────────┘
```

### 5.1 多 Agent 协作子流程

当任务需要多 Agent 协作时,L3.2 调度层按"专家矩阵"委派:
- 安全 Agent / 合规 Agent / 业务 Agent / 性能 Agent / 体验 Agent
- 各 Agent 独立上下文、独立工具集
- 协作通过"黑板"或"会议纪要"异步共享

L3.1 治理层为专家 Agent 设置**否决权限**——遇到红线问题,任何相关领域专家可一票否决,任务立即暂停升级。

## 8. 价值流对齐(Value Stream Alignment)(* zachshi 提出)

### 7.1 为什么需要价值流对齐?

当前 Agent 框架(包括太一)的核心缺陷:**只验证"任务是否完成",不验证"任务是否对最终业务目标有价值"**。

一个任务可能技术上 100% 成功:
- 代码提交了
- 验证通过了
- Skill 调用了

但它在**业务价值流**上可能是浪费:
- 提交了没人 review 的代码 → 价值流上是"库存积压"
- 生成的报表没人看 → 价值流上是"过量生产"
- 客服回复客户不要的答案 → 价值流上是"错误加工"

这些都不是 Agent 的"红线",但都是**对目标无价值的执行**。太一需要**价值流层**(H4)来识别和纠正这种"做对了事,但没做对方向"的问题。

### 7.2 三层目标体系(借鉴 APQC + OKR)

每个任务都应该能被追溯到三层目标之一或多个:

```
战略层(Strategic)      → 公司季度目标,如"Q3 交付质量提升"
  ↑
战术层(Tactical)        → 部门/项目 KPI,如"代码 review 平均响应 < 4h"
  ↑
任务层(Task)            → 单一 Agent 任务,如"提交今天的 feature 分支"
```

### 7.3 双模式:目标锰定(Goal Anchoring)

这是你最核心的设计输入:太一提供**两种目标锰定模式**,用户(管理员)二选一:

#### 模式 A:AI 推断 + 用户确认(动态交互模式)

适用场景:**不熟悉业务的新用户**,或**一次性任务**

工作流:
1. Agent 收到任务(如"提交今天的代码")
2. LLM 推断三层目标,生成候选:
   ```
   可能的服务目标:
   A. [任务] 提交 feature 分支
   B. [战术] 提升代码质量(假设)
   C. [战略] Q3 交付质量提升(假设)

   请选择(可多选,如 A+B, 或 A+C,或仅 A):
   ```
3. 用户在消息平台快速勾选(如"AB")
4. Agent 锁定目标栈,进入执行

**优点**:AI 帮用户想,用户只决策
**缺点**:多一轮交互,耗时略增

#### 模式 B:预设分级(自动许可模式)

适用场景:**熟悉业务的长期用户**,或**Cron/Heartbeat 主动调度**

工作流:
1. 管理员预设用户/角色/任务类型的"默认目标级别":
   ```
   helix.yaml:
   goal_anchoring:
     mode: preset
     default_stack: [tactical, strategic]   # 默认看到战术+战略
     overrides:
       - user: u1, scenario: dev.git:
           default_stack: [task]             # 开发者只关心任务层
       - user: ceo, scenario: ops.report:
           default_stack: [strategic]        # CEO 看战略层
   ```
2. Agent 收到任务 → 自动按预设装载目标栈 → 执行

**优点**:零交互,适合主动调度(Cron)
**缺点**:预设配置需要维护

### 7.4 价值流模板(Value Stream Templates)

为高频任务预定义端到端价值流(参考 APQC Process Classification Framework):

```
# 价值流示例:代码提交
stream: code_commit
layers:
  - { level: task,      goal: "提交 feature 分支" }
  - { level: tactical,  goal: "提升代码 review 通过率" }
  - { level: strategic, goal: "Q3 交付质量提升" }

# 价值流示例:客服回复
stream: customer_reply
layers:
  - { level: task,      goal: "回复客户咨询" }
  - { level: tactical,  goal: "提高首次解决率" }
  - { level: strategic, goal: "提升客户留存" }
```

### 7.5 增值评分(Value Contribution Scoring)

任务执行完成后,L4 验证层除"过没过红线"外,还输出"对目标贡献了多少":

```python
@dataclass
class ValueContribution:
    task_layer_contribution: float     # 0-1,任务层完成度
    tactical_alignment: float          # 0-1,对战术目标的贡献
    strategic_alignment: float         # 0-1,对战略目标的贡献
    wasted_steps: list[str]            # 价值流上的浪费环节
    notes: str
```

例如:
- 一个"提交 feature 分支"任务:
  - `task_layer_contribution: 1.0`(提交成功)
  - `tactical_alignment: 0.6`(提交了,但分支策略不符)
  - `strategic_alignment: 0.4`(提交了,但 review 响应慢,拖累 Q3 KPI)
  - `wasted_steps: ["git commit author 验证", "review request 等待"]`

### 7.6 瓶颈识别(Bottleneck Detection)

L5 迭代层持续统计 `ValueContribution` 汇聚:
- **任务高频浪费环节** → 优化 Skill / 减少不必要步骤
- **战术层低对齐的任务类型** → 提示 Skill 库需要新增
- **战略层低对齐的长期趋势** → 报告给管理员(可能场景定义/业务优先级错位)

### 7.7 嵌套成熟度模型(*zachshi 提出)

太一的成熟度评估不是单一分级的"系统级评分",而是**嵌套的多层级模型**——系统级、子系统级、Skill 级、Task 级各有自己的成熟度评估,且上下层级有累积性约束。

#### 7.7.1 嵌套结构与适用模型

| 层级 | 关注点 | 评估模型 | 太一对应 |
|---|---|---|---|
| **Task**(单次执行) | 输入输出是否正确 | Pass/Fail(单维度) | 单任务 L4 验证 |
| **Skill**(原子能力) | 流程是否标准化 | **CMMI / 华为 IPD** 过程域 | 每个 Skill 自带 quality_gate.md |
| **Subsystem**(子系统) | 是否可度量 | **CMMI L4 量化级** | 治理/调度/验证 各自有 KPI |
| **System**(整体) | 对外交付能力 | **DORA Metrics + Maturity** | 部署频率 / MTTR / 变更前置时间 |
| **Value Stream**(业务层) | 战略对齐 + 持续优化 | **IPD / APQC** 流程架构 | H4 价值流层 |

**为什么需要多模型?** 不同层级关注点不同——Task 看输入输出,Skill 看过程,Subsystem 看度量,System 看交付,Value Stream 看业务。**单一模型难以同时覆盖这些维度**。

#### 7.7.2 累积性原则(Cumulative)

**下层不达标,上层不能虚高**——这是 CMMI 的核心原则,嵌套到多层架构里仍然成立:

- L1 Skill → 系统不可能 L5
- L2 子系统 → 战略不可能优化级
- 每一层的"成熟度"是其下所有层级的"下确界"

**防 PPT 工程师**——不能写"我们的 Agent 是 L5 精英级",因为 Skill 是 L1。

#### 7.7.3 太一当前状态(2026-06-20)

| 层级 | 现状 | Phase 1 目标 |
|---|---|---|
| Task | L3(已实现单任务 Pass/Fail + L4 验证) | L3 |
| Skill | L2-L3(Demo 3 个 Skill 都带 quality_gate) | L3 + 部分 L4 |
| Subsystem | L2(治理/调度/验证 都有,但度量弱) | L3 |
| System | L1(Demo 阶段,未发布) | L2 |
| Value Stream | L1(尚未集成) | L2 |

#### 7.7.4 升级路径(Phase 1 重点)

**Skill L3 → L4 的升级条件**:
- ✅ quality_gate.md 完整
- ✅ 至少 3 个自动验证用例
- 🆕 必须接 H4 价值流 KPI(任务完成度 + 战术对齐率)
- 🆕 至少 5 次真实执行,贡献评分 ≥ 0.7

**Subsystem L2 → L3 的升级条件**:
- 🆕 治理层有可度量指标(决策延迟、拒绝率、误拦率)
- 🆕 调度层有可度量指标(模型路由准确率、Token 经济性)
- 🆕 验证层有可度量指标(验证耗时、误判率)

**System L1 → L2 的升级条件**(DORA):
- 部署频率 ≥ 1/月
- MTTR < 1 周
- 变更前置时间 < 1 天
- 变更失败率 < 20%

#### 7.7.5 为什么叫"嵌套"而不是"双轨"

原提的"双轨制"(Skill 用 IPD-CMMI + System 用 DORA)是**平行的**,意味着两者独立。

嵌套模型是**树状的**,意味着**上层受下层约束**——更精确地反映了系统本身的层级结构。这也是 zachshi 提出的核心洞察:"**体系是嵌套的,成熟度也是嵌套的**"。

#### 7.7.6 与 zachshi-ai 文章的关系

太一原文未提及成熟度模型——这是**为支持 H4 价值流层**,我提出的工程化衍生机制。但 zachshi 提出的"体系是嵌套的"这一思想为嵌套成熟度提供了哲学基础。

### 7.8 H4 与其他横切的关系

```
H1 记忆层 ←→ H4 价值流层     历史价值流数据可记忆/检索
H2 协议层 ←→ H4 价值流层     可暴露 MCP 工具:helix_value_stream_query
H3 观测层 ←→ H4 价值流层     ValueContribution 是观测数据的一部分
H4 价值流层 ←→ L1-L5         每层都需感知目标
```

### 7.9 Phase 1 优先级

按你建议的 B(完整补入),Phase 1 实现顺序:

1. **目标锰定语法**(TaskGoal 数据结构)—— 详见 Tech Doc §3.4
2. **预设分级配置文件**——YAML schema + 加载逻辑
3. **AI 推断目标**——LLM 提示词工程(消耗 LLM tokens,但仅在模式 A)
4. **价值流模板**——内置 5-10 个高频模板(代码/客服/报表等)
5. **增值评分**——L4 输出新增 ValueContribution 字段
6. **瓶颈识别**——L5 批处理任务,周报/月报

## 10. 风险与边界

### 8.1 适用范围
- ✅ 强适配:代码开发、运维自动化、合规审计、流程执行、数据处理
- ⚠️ 部分适配:知识检索、文档生成(需较多主观人审)
- ❌ 不适配:纯创意(文学、艺术、战略咨询)

### 8.2 已知风险

| 风险 | 影响 | 缓解 |
|---|---|---|
| 工程治理会拖慢 Agent | 用户可能嫌"慢" | 关键路径优化、缓存、人审并行 |
| 多 Agent 仲裁冲突 | 任务死锁 | 预设优先级 + 升级路径 |
| Skill 库膨胀 | 性能下降 | 质量门禁 + 定期清理 |
| 模型能力飞跃 | 工程治理价值衰减 | 显式设计成"可被替换的中间层",AGI 来时整套下沉 |
| 隐私/合规 | 触及红线 | 五层安全防线 + 凭证隔离 + SSRF 防护 |

### 8.3 未来演进
- **可能被吸收**:当模型能力进入"自带规则遵循 + 自带输出验证"阶段,L3 治理层、L4 验证层、L1 场景工程会被逐渐吸收进模型本身
- **长期保留**:L5 迭代层、记忆层(H1)、Skill 体系、协议层(H2)、观测层(H3),大概率仍是"基础设施"长期保留
- **时间预测**:这个吸收过程是渐进的(1-3 年),不会是突变;在过渡期内,**工程治理仍是决定生产可靠性的关键变量**
- **架构上的准备**:太一在 L3/L4 设计上明确"代码层治理"是临时措施,接口设计成可被替代的中间层——这避免了"沉没成本"陷阱

## 11. 总结:太一的核心承诺

> **太一不承诺 Agent 什么都能做,而承诺 Agent 在它能做的领域里——边界清晰、风险可控、持续进化。**

这是工业级数字员工的标准,也是 Agent 从"玩具"走向"生产工具"的分水岭。

**走得远的,永远是走得稳的。**
