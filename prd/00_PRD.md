## 1. 用户与场景

### 1.1 目标用户分层

| 层级 | 画像 | 关键需求 | 触达通道 |
|---|---|---|---|
| **个人开发者** | 全栈/独立开发者/SRE | "像 OpenClaw 一样本地跑,但要更可靠不会自动改我身份" | CLI / Web / Telegram / 飞书 |
| **技术团队** | 5-30 人研发/运维团队 | "项目级 Agent,统一管理 Skill/记忆/审计,可观测" | Web Console / API / 团队 IM |
| **企业部门** | 客服/运营/合规部门 | "多租户、权限隔离、审计合规、批量部署" | 私有化部署 / 企业微信 / 钉钉 |

### 1.2 典型场景(用 Job-to-be-Done 描述)

| 场景 | 用户的 Job | 太一 (The One) 提供的 Job |
|---|---|---|
| 代码提交 | "我想让 Agent 帮我跑测试 + commit + push,但绝不能改我身份" | 工具门禁:Git 操作前校验 committer 与 config 一致 |
| 定时报告 | "每天早上 8 点把昨天销售数据整理成 PDF 推到群里" | Cron 调度 + 数据安全场景 + 输出验证 + 多通道推送 |
| 客户邮件 | "帮我回 50 封咨询邮件,但敏感词要先过滤,语气要正式" | 场景工程(正式语气/敏感词过滤) + 同行 Agent 复评 |
| 跨平台监控 | "我的项目在 GitHub/Jira/Sentry/PagerDuty,Agent 能聚合告警再决策吗" | 多通道接入 + 多 Agent 协作 + 升级路径 |
| 研究报告 | "让 Agent 查 100 篇论文,沉淀为可复用的研究 Skill" | Skill 自生成 + 知识库更新 + 验证门禁 |

### 1.3 反例(明确不做的场景)

- ❌ 让 Agent 写小说/作诗
- ❌ 让 Agent 自主决定投资策略
- ❌ 让 Agent 模拟真人社交账号
- ❌ 让 Agent 自主与外部签订法律合同

## 2. 产品功能(按五层架构组织)

### 2.1 L1 输入/场景层

#### F1.1 场景管理
- **能力**:管理员可定义场景(行业/角色/任务类型),每个场景绑定约束规则
- **交互**:Web Console 可视化编辑;CLI 导入/导出场景模板
- **存储**:场景定义为 Markdown + YAML,人读可改
- **验收**:新增场景后,后续任务自动应用其约束

#### F1.2 知识库管理
- **能力**:支持导入规则、文档、合规条款;向量化语义检索 + FTS5 全文检索
- **形式**:支持 Markdown / PDF / Word / 网页抓取
- **验收**(注:以下指标是目标基线,实际需在 Phase 2 用 1000 文档量级语料跑出)
  - 检索延迟:P95 < 200ms (10万条)
  - 召回率(Recall@10):≥ 80%(基线:随机 30%,BM25-only 65%)
  - 准确率(Precision@5):≥ 70%(基线:随机 15%,BM25-only 50%)
  - 质量门禁:需有标准测试集(Phase 2 任务)

#### F1.3 任务信息注入
- **能力**:任务级一次性信息(临时参数、特殊要求)
- **隔离**:任务信息与长期知识严格分离,任务结束自动清理

### 2.2 L2 能力/资源层

#### F2.1 工具工程
- **能力**:统一工具注册中心,所有工具附带权限分级、输入输出规范、调用审计
- **形式**:工具以 `tool.json` + Python/TS 实现声明
- **内置工具**:文件读写、Shell 执行、HTTP 调用、SQL 查询、JSON 处理
- **验收**:所有高危工具必须经治理层审批才能调用

#### F2.2 技能工程(差异化核心)
- **能力**:Skill = 标准化流程 + 质量门禁 + 验证用例
- **四级分类**:
  - `bundled`(太一 (The One) 内置,如"Git 安全提交")
  - `managed`(社区策展,如"飞书周报生成")
  - `workspace`(组织私有,如"本公司客户回复模板")
  - `auto_generated`(Agent 自生成,需过门禁才能进入 managed)
- **验收**:每个 Skill 必带 `quality_gate.md`,无门禁的 Skill 不能进生产路径

#### F2.3 技能市场(Phase 3)
- **能力**:太一 (The One) Skill Hub,搜索/下载/评分/分享技能
- **机制**:基于 Markdown 的 Skill 描述,Git 协议分发

### 2.3 L3 调度/治理层(差异化核心)

#### F3.1 执行治理(中立裁判)
- **能力**:对调度层发出的执行申请,按规则返回"放行/拒绝/转人审"
- **规则类型**:
  - 红线规则(必拒,如"不得读取 ~/.ssh/id_rsa")
  - 场景约束(必审,如"高风险操作需人审")
  - 工具权限(分级,按用户/角色)
- **凭证隔离**:子进程只拿安全环境变量,API Key 严格过滤
- **SSRF 防护**:URL capable 工具强制白名单
- **验收**:红线违规 100% 被拦截(测试用例覆盖)

#### F3.2 任务适配调度(决策者)
- **能力**:基于任务类型 + 场景 + 知识,选择最优模型/工具/技能
- **智能路由**:否决式启发规则——任一"复杂"规则触发 → 主力模型
- **多 Agent 委派**:可调用 5 类专家 Agent(安全/合规/业务/性能/体验)
- **Cron / Heartbeat**:主动调度,at/every/cron 三种语法
- **验收**:简单任务自动路由到轻量模型,Token 成本降低 ≥ 30%

#### F3.3 治理-调度协议(关键接口)
- **形式**:REST / gRPC 接口
- **流程**:调度 → 申请许可证 → 治理审查 → 放行/拒绝
- **审计**:所有申请-决策-执行记录入审计日志,永久可追溯

### 2.4 L4 输出/验证层(差异化核心)

#### F4.1 客观验证清单
- **能力**:基于任务 + 场景,自动生成验证清单(JSON Schema、必含字段、合规项)
- **执行**:100% 自动化,失败立即回弹 L3
- **示例**:Git 提交后必校验 author/email 与 git config 一致

#### F4.2 主观人审兜底
- **能力**:对主观偏好/灰色判断,转交人审(消息平台通知)
- **超时**:人审超时自动按预设策略(默认通过/拒绝/延期)

#### F4.3 同行 Agent 复评
- **能力**:调起独立验证 Agent 复评(降低同源偏见)
- **触发**:对高风险任务或自评高分时强制复评

### 2.5 L5 迭代/优化层

#### F5.1 任务轨迹分析
- **能力**:每次任务产生结构化轨迹(决策、验证结果、用户反馈)
- **存储**:SQLite + FTS5,支持历史搜索

#### F5.2 Skill 自动沉淀
- **触发**:同一类任务完成 N 次后,自动生成 Skill 草稿
- **门禁**:草稿需走质量门禁才能进入 managed

#### F5.3 规则补丁建议
- **能力**:基于失败案例,OODA 循环建议规则补丁(待管理员审批)

### 2.6 横切:H1 记忆层

#### F6.1 五层记忆
- **L1 短期上下文**:会话内,自动清理
- **L2 技能库(SKILL.md)**:程序性知识
- **L3 向量索引**:语义检索
- **L4 Honcho 用户建模**:辩证画像(正-反-合)
- **L5 FTS5 全文**:历史对话可搜索

#### F6.2 记忆导出/导入
- **能力**:支持把记忆导出为 Markdown 归档,或从备份恢复
- **格式**:人类可读,可手动编辑

### 2.7 横切:H2 协议层

#### F7.1 MCP 双向
- **能力**:太一 (The One) 既是 MCP Client(消费外部工具),也是 MCP Server(暴露自身能力)
- **互操作**:Claude Code / Cursor / Codex 可直接调用 太一 (The One) 工具

#### F7.2 OpenAI 兼容 API
- **能力**:太一 (The One) Gateway 暴露 OpenAI 兼容的 `/v1/chat/completions` 接口
- **场景**:任何 OpenAI 客户端可无改造接入

#### F7.3 ACP 协议
- **能力**:支持 Agent Client Protocol,与 VS Code / Zed / JetBrains 集成

### 2.8 横切:H3 观测/审计层

#### F8.1 全链路追踪
- **能力**:OpenTelemetry 标准,每次执行可重放
- **存储**:Trace + Span + 属性

#### F8.2 Token 成本归因
- **能力**:每个任务、每个模型、每个工具调用的成本可视化
- **报表**:日/周/月/项目维度

#### F8.3 安全审计
- **能力**:所有高危操作、治理决策、人审结果永久留痕
- **查询**:Web Console + CLI 查询

### 2.9 入口与通道

#### F9.1 多通道接入
- **P0 通道**:CLI、Web Console
- **P1 通道**:飞书、钉钉、Telegram、Discord、Slack、WhatsApp
- **P2 通道**:企业微信、微信个人号、Teams、Signal、Email

#### F9.2 Gateway 单一入口
- **能力**:WebSocket / HTTP 统一入口
- **特性**:鉴权、限流、会话管理、状态广播

## 4. 用户故事(精选)

### US-001:开发者提交代码
> **作为** 开发者
> **我想要** 让 Agent 帮我跑测试、commit、push
> **以便于** 提高效率,但绝不能被改 committer

**验收**:
- Agent 提交前必须读取并使用 `git config user.name/user.email`,不可覆盖
- 提交后自动验证 GitHub 端 author 与本地一致
- 不一致则回弹 + 报警,绝不进入推送环节

### US-002:运营定时周报
> **作为** 运营经理
> **我想要** 每周一早上 8 点自动生成上周数据周报,推到飞书群
> **以便于** 团队周会前自动有材料

**验收**:
- Cron 表达式配置:`0 8 * * MON`
- 失败重试 3 次(指数退避)
- 周报 PDF 格式 + 推送到指定飞书群
- 任何一次执行可在 Web Console 回放完整轨迹

### US-003:客服回复
> **作为** 客服主管
> **我想要** Agent 帮助回复客户咨询,但涉及退款必须转人审
> **以便于** 提高效率且风险可控

**验收**:
- 检测到"退款/退货/投诉"关键词 → 场景工程自动加载"客服-退款"场景
- 场景约束:涉及金额 > 100 必须人审
- 治理层拦截,转人审,人审结果回写到知识库

### US-004:多 Agent 协作
> **作为** 项目经理
> **我想要** 让安全/合规/业务专家 Agent 一起审一份合同草稿
> **以便于** 多角度把关,避免单点偏见

**验收**:
- 三个专家 Agent 独立运行,产出独立意见
- 业务 Agent 提出的"建议采纳" → 优化维度,采纳决策归执行 Agent
- 合规 Agent 提出的"必须修改" → 红线维度,一票否决,任务暂停
- 升级路径:执行 Agent 接受 1 次复议,再否决则升级人审

### US-005:Skill 自生成
> **作为** 长期用户
> **我想要** Agent 把重复性任务自动沉淀为 Skill
> **以便于** 越用越快

**验收**:
- 同一类任务(检测:输入/输出模式相似)完成 5 次后,Agent 自动生成 Skill 草稿
- 草稿必带 `quality_gate.md` + 至少 3 个测试用例
- 草稿先进入沙箱库,跑通 10 次后管理员审批 → 进入 managed

### US-006:跨会话个性化
> **作为** 长期用户
> **我想要** Agent 记住我的偏好(不喜欢 emoji / 代码用 Tab / 报告用 Markdown)
> **以便于** 不用每次都重复交代

**验收**:
- Honcho 用户模型自动融合正-反-合
- 用户明确表达的偏好,优先级最高
- 偏好可视化,用户可查看/编辑/删除
- 新会话自动注入用户画像

## 6. 成功指标(KPIs)

### 6.1 用户价值
- **D7 留存率** ≥ 40%(7 天后仍活跃)
- **D30 留存率** ≥ 20%
- **NPS** ≥ 40

### 6.2 产品价值

**北极星指标(基于 zachshi 提出的 3 个新评测维度)**:

> 原文依据:"行业至今没有一套统一的 Agent 能力评估体系。AgentBench、GAIA,测的几乎全是'任务完成率'——能不能把事跑完,跑成了就算赢。**没有人测'规则遵从度',没有人测'隐性风险覆盖度',也没有人测'自主决策越界率'**。"

太一的产品价值锚定在 **zachshi 提出的三个新评测指标**(这些是 OpenClaw/Hermes/LangGraph 都没用的):

- **规则遵从度**(Rule Compliance Rate):Agent 执行过程中,有多少比例的规则被正确遵守(包含红线、场景约束、工具权限)
  - **目标**:≥ 99%(零漏过红线,场景约束 100% 触发,工具权限 100% 检查)
- **隐性风险覆盖度**(Implicit Risk Coverage):同一任务下,Agent 主动覆盖了多少用户隐性验收标准(权属、合规、安全、可维护性)
  - **目标**:≥ 80%(以 zachshi-ai 仓库上传场景为基线:能覆盖权属/敏感信息/分支策略等)
- **自主决策越界率**(Autonomous Decision Overreach Rate):Agent 自主决策中,有越界行为(未授权操作 / 修改用户配置 / 越过场景) 的比例
  - **目标**:0%(红线违规=0;场景约束越界=0;隐式修改用户身份信息=0)

**传统指标(辅助)**:
- **任务成功率**(客观验证通过率)≥ 85%
- **红线拦截率** = 100%(绝对值)
- **Token 成本** 相比"全主力模型"基线降低 ≥ 30%
- **Skill 调用占比** 占比逐步上升(目标:30% 任务由 Skill 完成)

### 6.3 价值流对齐 KPI(* zachshi 提出,Phase 1 实现)

**价值流贡献北极星**(除上述三项外,太一的独有指标):

| KPI | 定义 | 目标 | 测量方式 |
|---|---|---|---|
| **任务-战术对齐率** | 任务完成度 × 对战术 KPI 的贡献均值 | ≥ 0.7 | L4 ValueContribution.战术_alignment 汇聚 |
| **战术-战略对齐率** | 战术动作最终服务于战略目标的比例 | ≥ 0.6 | 长期跟踪,月度统计 |
| **价值流浪费步进率** | 任务中"冗余步骤 / 总步骤" | ≤ 15% | L5 bottleneck detection |
| **目标锰定覆盖度** | 进入太一的任务中,明确锰定了三层目标的比例 | ≥ 80% | L1 输入统计 |
| **预设分级使用率** | 模式 B (预设) 占总任务比例 | 趋势上升 | 用户行为统计 |

**目标锰定交互指标**:

| 模式 | 指标 | 目标 |

| 模式 A | LLM 推断准确率(用户未修改比例) | ≥ 60% (Phase 1), ≥ 80% (Phase 3) |

## 8. 竞品定位(差异化)

| 维度 | OpenClaw | Hermes | LangGraph | **太一 (The One)** |
|---|---|---|---|---|
| 定位 | 个人 AI 助手 | 自进化 Agent | 多 Agent 编排 | **生产级 Agent OS** |
| 治理 | 弱 | 中 | 中 | **强(铁律:治理/调度分离)** |
| 验证 | 弱 | 弱(自评) | 弱 | **强(独立 gate + 同行 Agent 复评)** |
| 场景工程 | 无 | 无 | 无 | **独立模块** |
| Skill 门禁 | 无 | 无 | 无 | **强制 quality_gate.md** |
| 多 Agent 仲裁 | 无 | 中央调度 | 编排图 | **红线一票否决 + 升级路径** |
| 通道 | 25+ | 12 | 无 | **P0-P2 分级** |
| 记忆 | 双层 | 五层 | 自配 | **五层 + Honcho** |
| 闭环学习 | 无 | 强 | 无 | **PDCA + OODA 双循环** |
| MCP | 部分 | 双向 | 部分 | **双向 + 协议中立** |
| 适用场景 | 个人 | 个人/团队 | 团队 | **个人/团队/企业** |

## 9. 验收与发布流程

- 每个 Phase 结束需产出"功能验收报告" + "性能测试报告" + "安全审计报告"
