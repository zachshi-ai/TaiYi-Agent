§ 01

三个根问题,三个解法

现有 Agent 框架的核心痛点不是"能力不够",而是"边界不清、风险不可控"。 太一不追求 Agent 什么都能做,而追求在它能做的领域里——边界清晰、风险可控、持续进化。

能力边界

流行回答:大模型能力。
太一 (The One) 回答: 模型 × 场景 × 治理 × 验证。 OpenClaw 自动改 commit 作者的事故证明:即使模型很强,缺乏工程治理的 Agent 仍会越界。

多 Agent 协作

流行回答:多 Agent 讨论=更好决策。
太一 (The One) 回答: 协作无门等于无协作。 专家 Agent 的意见"被听到"和"被执行"是两码事——中间必须有一扇治理的门。

产品价值锚点

流行回答:能多完成任务。
太一 (The One) 回答: 不是"能做什么",而是"边界是否清晰、风险是否可控"。 少功能但边界清晰的工具,用户敢用在生产。

§ 02

五层骨架 + 三条横切

纵向是五层关注点分离;横向是三条横切关注点贯穿。 核心创新在 L3——治理权与调度权强制分离。

§ 06

七条设计原则

INCOSE 风格的系统设计原则,贯穿 太一的所有工程决策。

P1
关注点分离 每一层只管一件事,五层互不越界
P2
单一真相源 Gateway / MEMORY.md / Skill Index 各管一摊
P3
显式优于隐式 所有决策必须带"理由+证据链",审计可重放
P4
渐进式降级 每层都有降级路径(模型降级→人工接管)
P5
边界即设计 明确"不做什么"和"做什么"一样重要
P6
闭环优于开环 PDCA + OODA 双循环,所有 Skill 使用后必评估
P7
制度优先于劝导 ⭐ 用代码约束行为,而不是用 prompt 说服模型。关键校验全部下沉到治理层,模型"看不到但绕不过"。

§ 08

嵌套成熟度模型

zachshi 提出:“体系是嵌套的,成熟度也是嵌套的。”太一不用一个总评分衡量系统,而是 5 个层级各自评估、上层受下层约束。不同层级适合不同评估模型——Task 看输入输出、Skill 看过程、Subsystem 看度量、System 看交付、Value Stream 看业务。

❌ 双轨制 (不采用)

原提议的 "Skill 用 IPD-CMMI + System 用 DORA" 是平行结构,两者独立。意味着 "系统 L5 但 Skill L1" 可同时存在 — 这不符合 zachshi “嵌套” 哲学。

✅ 嵌套模型 (采用)

5 个层级各自评估,上层 ≤ 下层。每一层的成熟度是其下所有层级的 "下确界"。能防 "PPT 工程师” — 写“我们是 L5 精英” 但 Skill 还是 L1。

4.1 规则作为数据(不是散文)

# 最小规则 Schema —— 规则作为可评审、可版本化、可测试的数据
id: git.author.identity_match
domain: authorship        # security | authorship | compliance | safety | optimization
severity: red_line        # red_line(拦截) | advisory(警告)
applies_to: ["git.commit", "git.push"]
trigger: pre_execution    # pre | post | step_gate
check:
  type: deterministic     # deterministic | external_tool | model_judge
  assert: "commit.author == local.git.config.user"
on_fail:
  action: block             # block | warn | request_confirmation
  message: "Commit author does not match the local Git identity."
precedence: 80             # 越高越优先(冲突时胜出)
owner: platform-security  # 责任人
audit: true              # 每次评估记录不可篡改日志

4.2 检查清单:选择而非生成

opus 关键洞察:自动生成一份完整的检查清单,其难度本质上与原初可靠性问题相当——递归难题。解法:选择,不是生成。

最高 确定性检查

字符串 / AST / 状态比较。用于作者匹配、schema、格式。无需模型。

高 外部权威

linter / type-checker / 测试套件 / 策略引擎 / 扫描器。独立维护。

最低 模型评判

仅限真的模糊(语气、设计质量)。需校准、采样、隔离独立实例。

4.3 多 Agent 闸门与仲裁

红线维度

数据安全 / 权属 / 核心规则。该领域专家有一票否决权。失败 → 任务暂停修正,不得前进。

优化维度

性能 / 体验 / 优雅度。专家仅建议,是否采纳归执行 Agent 或人审。

冲突裁决: (1) 比较 precedence 高者胜 → (2) 同级硬冲突 → fail closed + 升级人审 → (3) 每次仲裁记录,用于后续调优

4.4 落地优先级(不要一起起 5 层)

1
列出不可逆 / 高爆炸半径操作
写入 / 支付 / 身份 / 对外发布 / 删除。几乎所有实际损失都在这个列表里。
2
只为这些加确定性前置闸门
作者匹配是第一步。可靠性最大提升 × 最少工作量。
3
验证者从执行者拆出 + 不可篡改审计
P3 铁律落地。从“demo”跳到“可信任生产”的最小代价。
4
规则库 + 红线/建议拆分
§4.1 的 YAML Schema 全部使用上。
5
每任务检查清单 + 仲裁 / 升级
§4.2 选择而非生成 + §4.3 多 Agent 闸门。
6
闭环反哺
每次新失败反哺规则/Skill/知识。Loop Engineering。

4.7 反模式(可靠性气味)

  • ✗ 关键规则仅在 prompt 里表达为"记得要..."
  • ✗ 干活的组件同时签收(P3 违反)
  • ✗ 无不可篡改审计日志
  • ✗ "红线" 检查只 warn 不 block
  • ✗ 单一巨型模型评判 prompt 是唯一验证
  • ✗ 所有约束同一个严重级——无红线/建议区分
  • ✗ Agent 执行不可逆 / 破坏性操作无确认
  • ✗ 规则手册从未修剪/演进

4.8 闸门的成本——以及何时不用

闸门不免费
  • 增加延迟
  • 可能产生 false-block,侵蚀用户信任
  • 规则手册本身需要维护与漂移
正确用法
  • 硬、fail closed 仅用于不可逆 + 高爆炸半径
  • 低风险工作 使用建议检查 + 采样
  • 红线集小而高信号
  • 不要为探索性 / 创意 / 低风险任务加闸门
  • Loop Engineering 应该裁剪规则

目标是 校准的控制,不是 最大的控制。
